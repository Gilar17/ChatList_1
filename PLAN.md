# План реализации ChatList

Поэтапный порядок действий на основании [PROJECT.md](PROJECT.md) и [DATABASE.md](DATABASE.md).

---

## Этап 1. Подготовка окружения и структуры проекта

1. Организовать структуру проекта и зафиксировать модули:
   - `db.py` — работа с SQLite
   - `models.py` — логика моделей и промтов
   - `network.py` — HTTP-запросы к API
   - `main.py` — графический интерфейс PyQt
2. Обновить `requirements.txt`:
   - PyQt6
   - httpx (или requests)
   - python-dotenv
3. Добавить `.env.example` с примерами переменных API-ключей (`OPENROUTER_API_KEY`, `OPENAI_API_KEY` и т.д.).
4. Обновить `README.md`: установка зависимостей, настройка `.env`, запуск в PowerShell.
5. Проверить `.gitignore`: исключить `.env`, `chatlist.db`, `.venv`, `build/`, `dist/`.

**Результат:** проект подготовлен к разработке, зависимости и документация описаны.

---

## Этап 2. Реализация модуля базы данных `db.py`

1. Создать `db.py` — единственная точка доступа к SQLite.
2. Реализовать инициализацию БД: файл `chatlist.db`, создание таблиц по схеме из `DATABASE.md`.
3. Включить `PRAGMA foreign_keys = ON` при подключении.
4. Добавить CRUD-методы для таблиц:
   - `prompts` — создание, чтение, поиск
   - `models` — создание, чтение, обновление, удаление
   - `results` — запись и чтение сохранённых ответов
   - `settings` — чтение и запись настроек с `updated_at`
5. Реализовать получение активных моделей: `is_active = 1`.
6. Проверить создание и чтение записей простым скриптом или тестом.

**Результат:** схема БД работает, все операции доступны через `db.py`.

---

## Этап 3. Реализация модуля работы с моделями `models.py`

1. Описать структуру данных модели (dataclass или словарь) с полями из таблицы `models`.
2. Реализовать загрузку списка моделей и активных моделей из БД через `db.py`.
3. Реализовать чтение API-ключа из переменной окружения по полю `api_key_env_var`.
4. Добавить функции управления моделями:
   - добавление
   - редактирование
   - включение / отключение (`is_active`)
   - удаление
5. Подготовить начальный набор моделей (OpenAI, DeepSeek, Groq, OpenRouter и т.д.) через seed-данные.
6. Реализовать функции работы с промтами: сохранение, загрузка списка, получение по id.

**Результат:** модели и промты управляются через `models.py`, ключи читаются из `.env` по `api_key_env_var`.

---

## Этап 4. Реализация модуля сетевых запросов `network.py`

1. Реализовать базовую функцию отправки HTTP-запроса к API модели.
2. Использовать поля `api_url`, `api_id` (идентификатор модели в API) и ключ из `api_key_env_var`.
3. Добавить обработку ошибок: таймаут, неверный ключ, пустой ответ, HTTP-ошибки.
4. Реализовать адаптеры под разные типы API (`model_type`): минимум OpenAI-совместимый формат.
5. Добавить параллельную или последовательную отправку промта во все активные модели.
6. Вернуть унифицированный результат: имя модели, текст ответа или сообщение об ошибке.
7. **На этом этапе допустима заглушка:** функция возвращает тестовые ответы без реального обращения к API.

**Результат:** интерфейс отправки запросов готов; реальные или тестовые ответы возвращаются в едином формате.

## Этап 5. Реализация GUI-интерфейса `main.py` и временной таблицы результатов в памяти

1. Переработать текущее минимальное окно PyQt в каркас полноценного приложения.
2. Разделить окно на логические блоки:
   - многострочное поле ввода промта
   - кнопка «Отправить»
   - таблица временных результатов (модель, ответ, чекбокс `selected`)
   - кнопка «Сохранить»
3. Создать класс временной таблицы результатов **в памяти** (не в SQLite):
   - поля строки: `model_name`, `response_text`, `selected`
   - операции: очистка, добавление строк, получение выбранных (`selected = True`)
4. Применить стиль, похожий на стандартное Windows-приложение (Segoe UI, отступы, стиль `Windows11`).
5. Подключить заглушки: кнопка «Отправить» заполняет таблицу тестовыми данными из `network.py`.
6. Обеспечить запуск программы: `python main.py` открывает окно с основным интерфейсом.

**Результат:** программа запускается, показывает основной интерфейс и временную таблицу; обращения к нейросетям работают как заглушки.

## Этап 6. Интеграция модулей: ввод промта, отправка запросов, отображение и сохранение выбранных ответов

1. **Ввод и выбор промта:**
   - ввод нового промта в текстовое поле
   - выбор сохранённого промта из списка `prompts`
   - подстановка выбранного текста в поле ввода
   - сохранение нового промта в `prompts` при отправке (дата, текст, теги)
   - очистка временной таблицы при вводе нового промта
2. **Отправка запроса:**
   - по кнопке «Отправить» — взять текст промта
   - получить активные модели через `models.py`
   - отправить запросы через `network.py` (заменить заглушки на реальные вызовы)
   - заполнить временную таблицу ответами
   - показать индикатор загрузки на время ожидания
3. **Отображение результатов:**
   - таблица: колонки «Модель», «Ответ», чекбокс `selected`
4. **Сохранение выбранных результатов:**
   - по кнопке «Сохранить» — собрать строки с `selected = True`
   - записать в `results` через `db.py` (с `prompt_id`, `model_id`, копиями `prompt_text` и `model_name`)
   - очистить временную таблицу после успешного сохранения
   - показать сообщение об успехе или ошибке
5. Проверить полный сценарий: ввод промта → отправка → выбор ответов → сохранение → новый запрос.

**Результат:** все модули связаны, основной рабочий процесс из PROJECT.md реализован.

4. **Настройки** (`settings`):

5. **Опционально (из раздела 6 PROJECT.md):**
   - экспорт выбранных результатов в Markdown и JSON
   - расширение поддержки типов моделей (OpenAI, DeepSeek, Groq, OpenRouter)
   - логирование запросов в файл
   - поиск и сортировка во всех таблицах интерфейса

**Результат:** программа настраивается без правки кода, история и результаты доступны для просмотра и управления.

## Этап 10. AI-ассистент для улучшения промтов

Реализация функции из раздела 7 [PROJECT.md](PROJECT.md): отправка исходного промта в одну модель через существующий OpenRouter-клиент и показ пользователю улучшенных вариантов с возможностью подстановки в поле ввода.

### 10.1. Анализ и проектирование

1. Зафиксировать точки интеграции в текущем коде:
   - поле ввода промта и верхняя строка кнопок — `main.py`;
   - отправка запроса к одной модели — `network.send_prompt_to_model()`;
   - выбор модели и чтение ключа — `models.py`;
   - настройки приложения — таблица `settings`, диалог `SettingsDialog` в `dialogs.py`.
2. Определить формат ответа ассистента:
   - модель возвращает **структурированный текст** (предпочтительно JSON в markdown-блоке);
   - поля ответа:
     - `improved` — улучшенная версия промта;
     - `alternatives` — массив из 2–3 переформулировок;
     - `adaptations` — необязательный объект с вариантами под тип задачи (`code`, `analysis`, `creative`).
3. Подготовить системный промт ассистента (константа в отдельном модуле или в `network.py`):
   - инструкция вернуть только JSON без пояснений;
   - требование сохранить смысл и язык исходного промта;
   - ограничение длины альтернатив.
4. Выбрать модель по умолчанию для ассистента:
   - настройка `prompt_assistant_model_id` в `settings` (ID записи из таблицы `models`);
   - запасной вариант — первая активная модель из `load_active_models()`;
   - при отсутствии активных моделей или ключа — понятное сообщение пользователю.

**Результат:** согласован формат запроса/ответа и места изменений в коде без правки схемы БД (только новые ключи в `settings`).

---

### 10.2. Серверная логика (модули `network.py` и `models.py`)

1. В `models.py` добавить функции:
   - `get_prompt_assistant_model()` — получить модель ассистента по настройке или fallback;
   - `get_prompt_assistant_model_id()` / `set_prompt_assistant_model_id()` — чтение и запись настройки;
   - `parse_prompt_assistant_response(text: str) -> dict` — разбор JSON из ответа модели с обработкой ошибок формата.
2. В `network.py` добавить функцию:
   - `improve_prompt(original_prompt: str, model: ModelRecord, timeout: float) -> RequestResult`;
   - внутри — формирование системного и пользовательского сообщений;
   - вызов существующего `_send_openrouter()` / `send_prompt_to_model()` **без дублирования HTTP-кода**;
   - логирование запроса через `_log_request()` с пометкой «улучшение промта».
3. Обработать краевые случаи:
   - пустой исходный промт — ошибка до отправки;
   - ответ не JSON — показать сырой текст и сообщение «Не удалось разобрать ответ ассистента»;
   - таймаут и HTTP-ошибки — использовать существующие сообщения `RequestResult.error`.
4. Не менять логику массовой отправки промта в активные модели (`send_prompt_to_all`).

**Результат:** улучшение промта выполняется одним запросом к одной модели через текущий OpenRouter-клиент.

---

### 10.3. Фоновый поток и интеграция в `main.py`

1. Создать `ImprovePromptWorker(QThread)` по аналогии с `SendPromptWorker`:
   - вход: исходный текст промта;
   - выход: сигнал `finished(dict | str | None)` — разобранный результат или текст ошибки.
2. Добавить кнопку **«Улучшить промт»** в верхнюю строку рядом с «Отправить»:
   - та же ширина/шрифт, что у существующих кнопок;
   - неактивна при пустом поле ввода;
   - блокируется на время запроса (как «Отправить»).
3. Обработчик `on_improve_prompt()`:
   - взять текст из `prompt_input`;
   - проверить наличие модели ассистента и API-ключа;
   - запустить worker, показать индикатор ожидания (строка состояния или блокировка кнопок — в стиле текущего UI);
   - по завершении открыть диалог результатов.
4. Не очищать временную таблицу результатов при улучшении промта (это не новый запрос к нескольким моделям).

**Результат:** кнопка в главном окне запускает улучшение без блокировки интерфейса.

---

### 10.4. Диалог результатов (`dialogs.py`)

1. Создать `PromptAssistantDialog(QDialog)`:
   - заголовок: «AI-ассистент: улучшение промта»;
   - блок **«Исходный промт»** — read-only `QTextEdit`;
   - блок **«Улучшенный промт»** — read-only `QTextEdit` + кнопка «Подставить»;
   - блок **«Альтернативные варианты»** — список или несколько read-only полей (2–3 шт.) с кнопкой «Подставить» у каждого;
   - блок **«Адаптации»** (если есть в ответе) — подписи «Код», «Анализ», «Креатив» с кнопками «Подставить»;
   - кнопка **«Закрыть»** внизу справа.
2. Кнопка **«Подставить»**:
   - записывает выбранный текст в `prompt_input` главного окна;
   - закрывает диалог после подстановки;
   - не сохраняет промт в БД автоматически (сохранение — по существующей логике при «Отправить»).
3. Если разбор ответа не удался — показать сырой ответ модели и предложить скопировать вручную.
4. Стиль: Segoe UI, отступы и кнопки — как в `PromptEditDialog` / `SettingsDialog`.

**Результат:** пользователь видит исходный, улучшенный и альтернативные варианты и подставляет выбранный одним кликом.

**Результат:** пользователь может выбрать модель ассистента без правки `.env` и кода.

### 10.6. Тестирование

1. **Успешный сценарий:** непустой промт → «Улучшить промт» → получены `improved` и 2–3 альтернативы → «Подставить» заполняет поле ввода.
2. **Пустой промт:** кнопка неактивна или сообщение «Введите текст промта».
3. **Нет API-ключа:** понятное сообщение, приложение не падает.
4. **Неверный формат ответа модели:** показ ошибки разбора, интерфейс остаётся рабочим.
5. **Таймаут:** сообщение об ошибке, повторный запуск возможен.
6. **Регрессия:** отправка промта в несколько моделей, сохранение результатов, окна управления промтами/моделями работают как раньше.
7. Проверить логи `chatlist.log` — ключи не попадают в файл.

**Результат:** функция стабильна, основной сценарий ChatList не нарушен.
